  <name>Task 1: Wire PathExtender into maintain_and_evaluate_frames with event dispatch</name>
  <files>src/engine.rs</files>
  <action>
Modify `maintain_and_evaluate_frames` in engine.rs to accept the event and dispatch to incremental path extension for EdgeAdded events, falling back to full rematerialize for all other event types.

**Step 1: Update function signature.**

Change:
```rust
fn maintain_and_evaluate_frames(
    frames: &[(u64, Arc<RwLock<Frame>>)],
    graph: &Graph,
    epoch: Epoch,
    prev_deltas: &HashMap<u64, i64>,
) -> Vec<(u64, i64)>
```
To:
```rust
fn maintain_and_evaluate_frames(
    frames: &[(u64, Arc<RwLock<Frame>>)],
    graph: &Graph,
    epoch: Epoch,
    prev_deltas: &HashMap<u64, i64>,
    event: &Event,
) -> Vec<(u64, i64)>
```

**Step 2: Add event-based dispatch inside the scoped thread body.**

Replace the unconditional `frame.rematerialize(graph, epoch);` with:

```rust
match event {
    Event::EdgeAdded { source, target, type_id, .. } => {
        // Incremental: compute new paths via PathExtender
        let deltas = crate::path_extender::extend_edge_added(
            frame.anchor(),
            frame.pattern(),
            graph,
            *source,
            *target,
            *type_id,
        );
        for path in deltas.new_paths {
            frame.apply_delta(path, epoch, Delta(1));
        }
    }
    _ => {
        // Fallback: full re-traverse for non-EdgeAdded events
        frame.rematerialize(graph, epoch);
    }
}
```

**Step 3: Update both call sites to pass the event.**

There are exactly 2 call sites for `maintain_and_evaluate_frames`:

1. **`ingest()` method (~line 364):** Currently passes `(&affected_frames, graph_ref, epoch, prev_deltas)`. Add `&event` as the 5th argument. The `event` is already available in `ingest()` scope (it was cloned at the top for the ring buffer push, and the original borrows are available from `&event` in the match arms).

2. **`flush_coalescer()` method:** This batches multiple events and doesn't have a single original Event. Per Pitfall 7 in the research, the coalescer path should continue using full rematerialize as fallback. Create a synthetic `Event::NodeRemoved { node_id: NodeId(0) }` or better: extract the maintain_and_evaluate_frames call in flush_coalescer to pass a non-EdgeAdded event so it always hits the fallback branch. The simplest approach: pass `&Event::NodeRemoved { node_id: NodeId(0) }` as a "force-rematerialize" sentinel. Alternatively, create a small wrapper or just inline the rematerialize call for the coalescer path. **Preferred approach:** The coalescer path already has its own call. Pass any non-EdgeAdded event variant to trigger the fallback. Use `Event::PropertyChanged { node_id: NodeId(0), key: 0, value: PropertyValue::Int(0) }` as a sentinel -- it will always hit the `_ => frame.rematerialize()` fallback branch. Add a comment: `// Coalescer batches multiple events -- always rematerialize (incremental dispatch deferred)`

**Step 4: Ensure `Delta` is imported in scope.** The match arm uses `Delta(1)`. Verify `use crate::types::Delta` is in the engine.rs imports. It should already be there since engine.rs uses `Delta` elsewhere, but confirm.

**Borrow checker note (Pitfall 6):** The `&Event` reference passed to `maintain_and_evaluate_frames` must outlive the `std::thread::scope`. Since `event` is borrowed from the `ingest` parameter (or from the cloned `event` in the ingest body), it lives on the stack of `ingest` and therefore outlives the scope. No lifetime issues expected.

Add new oracle tests to the `#[cfg(test)]` section of engine.rs that specifically validate incremental EdgeAdded behavior. These tests use the existing `oracle_check()` function which compares maintained frame state against fresh DFS rematerialization.

**New tests to add (at least 4):**

1. **`test_oracle_incremental_two_hop_edge_added`**: Build graph: nodes 1(10), 2(20), 3(30). Edge 1->2 type 100. Register 2-hop frame: anchor=1, hop0=(Out/100/type20), hop1=(Out/200/type30). oracle_check (0 paths). Then add edge 2->3 type 200. oracle_check must pass -- incremental PathExtender should produce [1,2,3] matching what full DFS would find. Assert `query_frame` returns exactly 1 path: [1,2,3].

2. **`test_oracle_incremental_multiple_edge_adds`**: Build graph: nodes 1(10), 2(20), 3(20). Register 1-hop frame: anchor=1, hop0=(Out/100/type20). Add edge 1->2 type 100. oracle_check. Add edge 1->3 type 100. oracle_check. Assert frame has 2 paths: [1,2] and [1,3]. This tests that multiple sequential incremental EdgeAdded events accumulate correctly.

3. **`test_oracle_incremental_edge_added_then_removed`**: Build graph: nodes 1(10), 2(20). Register 1-hop frame. Add edge 1->2 type 100. oracle_check (1 path). Then remove the edge. oracle_check (0 paths -- removal still uses full rematerialize fallback). This verifies that the incremental +1 state is correctly wiped by a subsequent rematerialize (evict+DFS) on EdgeRemoved.

4. **`test_oracle_incremental_edge_added_no_match`**: Build graph: nodes 1(10), 2(20). Register 1-hop frame with edge_type=Some(TypeId(100)). Add edge 1->2 type 999 (doesn't match pattern). oracle_check -- frame should have 0 paths. Assert query_frame returns empty. This tests that non-matching edges produce no spurious paths.

5. **`test_oracle_incremental_three_hop_middle_edge`**: Build graph: nodes 1(10), 2(20), 3(30), 4(40). Edges: 1->2 type 100, 3->4 type 300. Register 3-hop frame: anchor=1, hop0=(Out/100/type20), hop1=(Out/200/type30), hop2=(Out/300/type40). No complete path yet. Add edge 2->3 type 200 (satisfies hop 1). oracle_check must pass -- incremental PathExtender should produce [1,2,3,4] by finding backward prefix [1,2], appending 3, then forward extending to 4 via type 300. Assert 1 path.

All tests follow the same pattern as existing oracle tests: build graph via ingest, register frame, ingest more edges, call oracle_check after each mutation. The oracle_check function already handles the comparison.
