Create `src/set_trie.rs` implementing a trie over sorted sets:

**SetTrie struct:**
- Internal node structure: `SetTrieNode { children: HashMap<u64, SetTrieNode>, values: HashSet<u64> }` where keys are element IDs (NodeId.0 or encoded (NodeId, TypeId) pairs) and values are frame IDs.
- Root is a `SetTrieNode` with empty values.

**Public API (SETTRIE-01):**
- `pub fn new() -> Self` — create empty trie
- `pub fn insert(&mut self, elements: &[u64], value: u64)` — insert a set (pre-sorted ascending) associated with a value (frame_id). Walk/create trie path for each element, store value at terminal node.
- `pub fn remove(&mut self, elements: &[u64], value: u64)` — remove a value from a set path. Walk trie, remove value from terminal. Prune empty nodes.
- `pub fn query_containing(&self, elements: &[u64]) -> HashSet<u64>` — find all values whose registered set CONTAINS all given elements. Walk each element path, collect values at nodes reachable by all query elements. Use recursive DFS: at each node, collect values that appear in ALL element subtrees.
- `pub fn query_intersecting(&self, elements: &[u64]) -> HashSet<u64>` — find all values whose registered set INTERSECTS with the given elements (shares at least one element). Union of values along each element's path.

**Implementation notes:**
- Elements must be sorted before insertion/query for trie ordering.
- For `query_containing`: collect frame IDs that exist at the end of paths containing ALL query elements. This is the intersection of posting lists for each element.
- For `query_intersecting`: collect frame IDs that exist at the end of paths containing ANY query element. This is the union of posting lists.
- Include `pub fn len(&self) -> usize` to count total unique values stored.
- Implement Default trait via delegation to new().

**Tests (co-located in set_trie.rs, TEST-25 partial):**
- `test_insert_and_query_containing` — insert 5 sets, verify containment queries return correct frame IDs
- `test_query_intersecting` — verify union semantics
- `test_remove` — insert, remove, verify absent from queries
- `test_empty_trie` — all queries on empty trie return empty
- `test_duplicate_insert` — inserting same set twice with same value is idempotent

Create `src/count_min_sketch.rs` implementing a fixed-size probabilistic counter (CMS-01):

**CountMinSketch struct:**
- `width: usize` (default: 1024)
- `depth: usize` (default: 4)
- `matrix: Vec<Vec<u64>>` (depth x width)
- `seeds: Vec<u64>` (one per row, for hash functions)

**Public API:**
- `pub fn new(width: usize, depth: usize) -> Self` — allocate matrix, generate deterministic seeds (use row index * prime for reproducibility)
- `pub fn increment(&mut self, key: u64)` — for each row, compute hash = (key.wrapping_mul(seed).wrapping_add(seed >> 16)) % width, increment matrix[row][col]
- `pub fn estimate(&self, key: u64) -> u64` — return minimum across all rows for the hashed columns (minimum is closest to true count since it has least collision)
- `pub fn reset(&mut self)` — zero all counters

**Implementation notes:**
- Hash function per row: use `key.wrapping_mul(seeds[row]).wrapping_add(seeds[row] >> 16) % width` for each depth row.
- Seeds: `[0x9e3779b97f4a7c15, 0x517cc1b727220a95, 0x6c62272e07bb0142, 0x62b821756295c58d]` (well-known constants, truncate/extend for non-default depth).
- No underestimate guarantee: increment always adds to all rows, estimate takes minimum.
- Implement Default with width=1024, depth=4.

**Tests (co-located, TEST-27 partial):**
- `test_basic_increment_estimate` — increment key 5 times, estimate returns >= 5
- `test_no_underestimate` — increment N times, estimate >= N always
- `test_reset_clears` — after reset, all estimates are 0
- `test_different_keys_independent` — two keys with different counts return different estimates

**Update `src/routing.rs` (SETTRIE-02):**

Replace the HashMap-based posting lists with SetTrie while keeping the SAME public API:

1. Replace `node_to_frames: HashMap<NodeId, HashSet<u64>>` and `edge_key_to_frames: HashMap<(NodeId, TypeId), HashSet<u64>>` with a SINGLE `SetTrie` that stores frame IDs indexed by their element sets.

However, the Set-Trie must support the same query patterns. The key insight: keep the two HashMaps for register/unregister (they're the source of truth), but ADD a SetTrie as an acceleration structure for `affected_frames` queries.

Actually, the simpler and correct approach for SETTRIE-02: replace the internal HashMap posting lists with the SetTrie for node lookups:
- `node_trie: SetTrie` — maps individual NodeId elements to frame IDs
- `edge_key_to_frames: HashMap<(NodeId, TypeId), HashSet<u64>>` — keep HashMap for edge keys (Set-Trie is for set containment, edge keys are individual lookups)

Wait -- re-reading the requirement: "Inverted index uses Set-Trie internally while keeping same public API". The Set-Trie advantage is for routing queries where we need to find frames whose registered node-set CONTAINS or INTERSECTS with the event's affected nodes.

**Correct approach:**
- Replace `node_to_frames: HashMap<NodeId, HashSet<u64>>` with `SetTrie` for the node posting list.
- When `register_frame(frame_id, node_ids, edge_keys)` is called, insert the sorted node_ids set into the SetTrie with value frame_id. Also keep edge_key_to_frames HashMap.
- When `affected_frames(event)` is called, use `node_trie.query_intersecting(&[node_id_elements])` to find frames that share any of the event's affected nodes. Union with edge_key lookup results.
- `unregister_frame` calls `node_trie.remove(sorted_node_ids, frame_id)`.
- For `affected_frames_by_node(node_id)`, use `node_trie.query_intersecting(&[node_id.0])`.

But we need a mapping from frame_id -> its registered node set for unregister. Add: `frame_nodes: HashMap<u64, Vec<u64>>` to store the sorted element set per frame.

The SetTrie `query_intersecting` with a single element effectively returns all frame IDs that have that element in their set -- this is equivalent to the HashMap posting list lookup.

**Implementation changes to InvertedIndex:**
- Replace `node_to_frames: HashMap<NodeId, HashSet<u64>>` with `node_trie: SetTrie` and `frame_nodes: HashMap<u64, Vec<u64>>`.
- `register_frame`: convert node_ids to sorted `Vec<u64>`, insert into node_trie, store in frame_nodes.
- `unregister_frame`: look up frame_nodes[frame_id], call node_trie.remove with that set.
- `collect_by_node`: use `node_trie.query_intersecting(&[node_id.0])` to get frame IDs.
- `frame_count`: count unique frame IDs across node_trie and edge_key_to_frames.
- Keep ALL existing tests passing -- the public API does not change.

**Update `src/tiering.rs` (CMS-02):**

The frame prioritizer currently takes `query_count: u64` and `mutation_count: u64` as direct parameters. CMS-02 says "Frame prioritizer uses Count-Min Sketch INSTEAD OF per-frame query counters."

Add a `FrameActivityTracker` struct that wraps two CountMinSketches (one for queries, one for mutations) and becomes the PRIMARY scoring interface:
- `pub struct FrameActivityTracker { query_sketch: CountMinSketch, mutation_sketch: CountMinSketch }`
- `pub fn new() -> Self` — creates with default 1024x4 sketches
- `pub fn record_query(&mut self, frame_id: u64)` — increments query sketch
- `pub fn record_mutation(&mut self, frame_id: u64)` — increments mutation sketch
- `pub fn estimated_query_count(&self, frame_id: u64) -> u64` — returns query sketch estimate
- `pub fn estimated_mutation_count(&self, frame_id: u64) -> u64` — returns mutation sketch estimate
- `pub fn priority_score(&self, frame_id: u64, epochs_since_last: u64, config: &TierConfig) -> f64` — calls the free function `priority_score()` using CMS-estimated counts from sketches instead of per-frame counters
- `pub fn reset(&mut self)` — resets both sketches (for epoch window rotation)

The existing `priority_score` free function's SIGNATURE is unchanged (it still accepts raw u64 counts), but the ENGINE must now obtain those counts from FrameActivityTracker rather than from per-frame counters. Plan 13-02 Task 3 handles the engine.rs integration that wires FrameActivityTracker into the engine's ingest pipeline and scoring call sites.
