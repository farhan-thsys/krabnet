  <action>
Create `src/path_extender.rs` as a new module. Add `pub mod path_extender;` to `src/lib.rs` (after `pub mod mcp;`) and a re-export line in the re-exports section: `pub use path_extender::{extend_edge_added, EdgeAddedDeltas};`.

The module implements a stateless function `extend_edge_added(anchor, pattern, graph, source, target, edge_type) -> EdgeAddedDeltas` that computes new complete paths produced by a newly added edge. The algorithm:

1. **`edge_matches_hop(hop, source, target, edge_type, graph) -> bool`**: Checks if a new edge could satisfy a given HopSpec. Verifies edge_type filter, direction compatibility, target_type filter, and property filter. For `Direction::Outgoing`, the reached node is `target`. For `Direction::Incoming`, the reached node is `source`. For `Direction::Any`, check both orientations and return true if either works.

2. **`backward_prefixes(anchor, pattern, graph, hop_idx, required_end) -> Vec<Vec<NodeId>>`**: For hop K, finds all partial paths from anchor through hops 0..K-1 that end at `required_end` (the edge's origin node). Special case: hop_idx == 0 means the backward prefix is `[anchor]` if `anchor == required_end`, else empty. For hop_idx > 0, runs `partial_dfs` from anchor through intermediate hops. The DFS filter logic MUST replicate `Frame::dfs_collect` exactly: direction, edge_type, target_type, property filter (Filter::None, Filter::PropertyEquals, Filter::HasProperty).

3. **`forward_extend(graph, current_path, reached_node, pattern, hop_idx, results)`**: After backward prefix + new edge's reached node is appended, continues DFS from `reached_node` through hops K+1..N-1. If hop_idx was the last hop, the path is already complete. Otherwise, runs `forward_dfs` replicating the same filter logic as dfs_collect. All complete paths (length == pattern.len() + 1) are pushed to results.

4. **`extend_edge_added(anchor, pattern, graph, source, target, edge_type) -> EdgeAddedDeltas`**: Main entry point. Returns empty if pattern is empty. Iterates each hop, calls `edge_matches_hop`. For matching hops with Direction::Outgoing: origin=source, reached=target. For Direction::Incoming: origin=target, reached=source (per Pitfall 1 -- incoming hop at node N follows edges where N is the target, reaching the source). For Direction::Any: try both orientations separately. Collects all backward prefixes, extends forward, deduplicates using `HashSet<Vec<NodeId>>`, returns `EdgeAddedDeltas { new_paths }`.

**Critical correctness notes:**
- Filter logic in backward_prefixes and forward_dfs MUST be identical to Frame::dfs_collect (lines 140-192 in frame.rs): `graph.neighbors(node, hop.direction, hop.edge_type)`, then check `hop.target_type` via `graph.get_node_type()`, then match on `hop.filter` for PropertyEquals/HasProperty.
- Path deduplication: Use `HashSet<Vec<NodeId>>` to remove duplicates before returning (Pitfall 4 from research).
- The graph already contains the new edge when this function runs (Step 2 mutation happens before Step 4 maintenance). This is correct -- backward prefixes using the post-mutation graph will find paths through the new edge at earlier hops too, which is valid behavior matching full DFS rematerialize.

**Struct:**
```rust
pub struct EdgeAddedDeltas {
    pub new_paths: Vec<Vec<NodeId>>,
}
```

**Test scenarios to implement (minimum 6 tests):**

1. **`test_single_hop_outgoing_edge_added`**: Graph with nodes 1(type 10), 2(type 20). Register a 1-hop outgoing pattern (edge_type=Some(100), target_type=Some(20)). Add edge (1->2, type 100). Call `extend_edge_added(anchor=1, pattern, graph, source=1, target=2, edge_type=100)`. Expect: `new_paths = [[1, 2]]`.

2. **`test_two_hop_second_edge_added`**: Graph with nodes 1(10), 2(20), 3(30). Edge 1->2 type 100 already exists. Pattern is 2-hop: hop0=(Out, type 100, target_type 20), hop1=(Out, type 200, target_type 30). Add edge 2->3 type 200. Expect: `new_paths = [[1, 2, 3]]`. This tests backward prefix at hop_idx=1 (partial DFS from anchor=1 through hop 0, ending at node 2).

3. **`test_incoming_direction`**: Graph with nodes 1(10), 2(20). Pattern is 1-hop with Direction::Incoming, edge_type=Some(100), target_type=Some(20). Add edge 2->1 type 100. For incoming hop: origin=target=1, reached=source=2. Backward prefix should be [1] (anchor=1 == origin=1). Forward: reached=2 is appended. Expect: `new_paths = [[1, 2]]`.

4. **`test_no_matching_hop`**: Graph with nodes 1(10), 2(20). Pattern is 1-hop (edge_type=Some(999)). Add edge 1->2 type 100. Edge type doesn't match. Expect: empty new_paths.

5. **`test_empty_pattern`**: Call with empty pattern. Expect: empty new_paths.

6. **`test_multi_hop_diamond_dedup`**: Graph with nodes 1(10), 2(20), 3(20), 4(30). Edges: 1->2 type 100, 1->3 type 100. Both nodes 2 and 3 connect to 4 via type 200. Pattern: 2-hop (Out/100/type20, Out/200/type30). Add edge 3->4 type 200. This only generates path [1,3,4] since only paths through the new edge are produced. Expect: `new_paths = [[1, 3, 4]]`. (If edge 2->4 type 200 also exists in the graph, extend_edge_added should NOT return [1,2,4] because that path doesn't traverse the new edge 3->4.)

7. **`test_property_filter`**: Graph with nodes 1(10), 2(20) where node 2 has property key=5 value=Int(42). Pattern: 1-hop (Out, type 100, target_type 20, Filter::PropertyEquals { key: 5, value: Int(42) }). Add edge 1->2 type 100. Expect: `new_paths = [[1, 2]]`. Then test with node 2 having wrong property value and verify empty result.

Each test builds a `Graph` directly (via `Graph::new()`, `add_node`, `add_edge`, `set_property`), constructs the pattern, and calls `extend_edge_added` to verify returned paths match expected.
