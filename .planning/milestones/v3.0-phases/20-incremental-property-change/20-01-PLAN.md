1. **PropertyChangedDeltas struct** (public, derives Debug):
   - `retracted_paths: Vec<Vec<NodeId>>` -- paths to retract as -1 deltas
   - `new_paths: Vec<Vec<NodeId>>` -- paths to assert as +1 deltas

2. **`pub fn reevaluate_property_changed()`** with signature:
   ```rust
   pub fn reevaluate_property_changed(
       anchor: NodeId,
       pattern: &[HopSpec],
       graph: &Graph,
       current_paths: &[&Vec<NodeId>],
       changed_node: NodeId,
   ) -> PropertyChangedDeltas
   ```

   Algorithm:
   a. **Early exit:** If pattern is empty or no hop has a property filter (all Filter::None), return empty deltas immediately. This is a significant performance optimization since most frames may have no property filters.

   b. **Step 1 -- Retract newly-invalid paths:** For each path in `current_paths`, for each hop K in pattern, check if `path[K+1] == changed_node`. If yes and hop K has a property filter (PropertyEquals or HasProperty), re-evaluate the filter against the node's current properties via `graph.get_property()`. Also check target_type if present via `graph.get_node_type()`. If the full hop constraint no longer passes, collect the path for retraction and break (one failing hop invalidates the whole path). Deduplicate retracted paths via HashSet.

   c. **Step 2 -- Assert newly-valid paths:** For each hop K with a property filter where `changed_node` satisfies the full hop constraint (target_type + property filter, checked via a `node_passes_hop()` helper):
      - Find "origin" nodes connected to `changed_node` matching the hop's direction and edge_type using a `find_hop_origins()` helper that queries `graph.neighbors()` with reversed direction (Outgoing: check incoming neighbors of changed_node; Incoming: check outgoing; Any: both).
      - For each origin, call the existing `backward_prefixes()` function to find partial paths from anchor through hops 0..K-1 ending at that origin.
      - For each prefix, call the existing `extend_forward()` or equivalent to complete the path through `changed_node` and remaining hops K+1..N-1.
      - Deduplicate new paths via HashSet.
      - Remove paths already in `current_paths` (avoid double-assertion -- this is critical per Pitfall 1 from research).
      - Also remove paths in the retracted set (defensive, theoretically unreachable since a path cannot both fail and pass the same filter).

3. **Helper: `fn node_passes_hop()`** (private):
   - Checks target_type (if Some, verify `graph.get_node_type(node_id) == Some(target_type)`)
   - Checks property filter (PropertyEquals: `graph.get_property(node_id, key) == Some(value)`; HasProperty: `graph.get_property(node_id, key).is_some()`; None: true)
   - Does NOT check edge_type (that is verified by the edge, not the node)

4. **Helper: `fn find_hop_origins()`** (private):
   - For Outgoing hop: origin has outgoing edge to changed_node, so query `graph.neighbors(changed_node, Direction::Incoming, hop.edge_type)` to find origins
   - For Incoming hop: origin follows incoming edge, reached_node is the source, so query `graph.neighbors(changed_node, Direction::Outgoing, hop.edge_type)` to find origins
   - For Any: query both directions, deduplicate

5. **Update module doc comment** to include `reevaluate_property_changed` in the module-level documentation.

6. **Update src/lib.rs** re-exports to include `reevaluate_property_changed` and `PropertyChangedDeltas`:
   ```rust
   pub use path_extender::{
       extend_edge_added, retract_edge_removed, retract_node_removed,
       reevaluate_property_changed,
       EdgeAddedDeltas, EdgeRemovedDeltas, NodeRemovedDeltas,
       PropertyChangedDeltas,
   };
   ```

IMPORTANT: Reuse the existing `backward_prefixes()` and `extend_forward()`/`forward_dfs()` from the same module (Phase 18). Do NOT hand-roll new DFS logic. Reuse the existing `edge_matches_hop_directed()` helper where applicable for filter evaluation. The `node_passes_hop()` helper is new because it checks the node-side constraints (target_type + property filter) without needing an edge context.

IMPORTANT: The graph has ALREADY been mutated by the time this function runs (engine.rs Step 2 calls `graph.set_property()` before `maintain_and_evaluate_frames`). `graph.get_property()` returns the NEW value. Do NOT try to capture old values.

IMPORTANT: For the backward_prefixes call, the existing function signature is `backward_prefixes(anchor, pattern, graph, hop_idx, origin)` where `hop_idx` is the hop index and `origin` is the node at position `hop_idx` in the path (the node from which the hop traverses). The `extend_forward` call continues from the reached node (changed_node) through remaining hops. Examine the existing extend_edge_added implementation to understand the exact calling convention.
